Using HTTPS

For security reasons, the local Hue API has to be accessed using HTTP over TLS (HTTPS) on port 443. This section explains how to use HTTPS specifically for the local API.

Hue Bridge Root CA

Signify has issued two private root CA Certificates used for Hue Bridges: one that is currently in use, and one secondary which we may switch to in the future. To use HTTPS correctly, the client must validate the Hue Bridge certificate against these root CA certificates. This is typically done by adding the CA certificate bundle into the key store trusted by the HTTP client, or for some HTTP clients by adding it as an input parameter when setting up the HTTPS connection. Common name validation

The other part of the certificate validation is checking that the Subject Common Name (CN) on the certificate matches the bridge id of the bridge you are trying to connect to. This is typically done by using the bridge id as hostname and adding a custom resolver that resolves it to the IP address, or by injecting a custom hostname verifier into the HTTPS client which checks that the server principal name matches the bridge id.

CURL Example

In this example it is assumed that the CA certificate bundle above is stored in a file called huebridge_cacert_bundle.pem. Furthermore the <ipaddress>, <hue-application-key> and <bridgeid> should be replaced by the actual values for the respective bridge.

First, this is an example curl command line request to retrieve the list of lights if it could be done using plain HTTP:

curl http://<ipaddress>/api/<hue-application-key>/lights

The equivalent request on HTTPS would then be:

curl --cacert huebridge_cacert_bundle.pem --resolve "<bridgeid>:443:<ipaddress>" https://<bridgeid>/api/<hue-application-key>/lights

Disabling certificate validation (e.g. in the curl example using the --insecure command line argument) can be done for development/testing purposes, but must never be used in production.

Intermediate certificates

By inspecting the certificate chain on a Hue Bridge

openssl s_client -showcerts -connect <ipaddress>:443

it can be seen that currently the device certificate is signed directly by the root certificate. In the future – in particular when we would switch to the secondary root certificate – the device certificates will likely be signed by an intermediate certificate which is in turn signed by the root. You still only have to bundle the root certificates with your client application, since the Hue Bridge would present both its device certificate and intermediate certificate during the TLS handshake. However, care should be taken that the HTTPS library you are using has support for certificate chains with intermediate certificates.

Self-signed certificates

Historically, a small subset of early produced Hue Bridges still used self-signed certificates. These bridges have since been updated to use certificates signed by Signify’s private Hue Bridge CA. If you run into a certificate validation error on an authentic Hue Bridge using the above guideline, then ensure you have automatic update enabled and/or manually update the bridge by connecting to it with the Hue app. If your app previously contained logic to handle these self-signed certificates by a “trust on first use” policy, then it is important to handle the case where the self-signed certificate is replaced by a properly signed certificate (and additionally the logic to handle self-signed certificates can be removed to simplify your app).

Web Browser

Web browsers do not have access to the root certificate of the Hue Bridges. This means that in case you would access the bridge using a browser (e.g. going to https://<ip-address bridge>) the browser will complain that the connection is not secure due to the fact that the certificate has not been certified by a trusted third party, and will ask you whether you want to continue and trust the certificate.

Performance

For performance reasons it is important to make sure consecutive requests within a short timeframe share the same connection by keeping the connection alive. This is typically supported by proper HTTP client libraries. The Hue Bridge also supports stateless session resumption with TLS session tickets, which we prefer clients to use to quickly resume TLS sessions on consecutive connections.

Hue EDK

If you are using the Philips Hue EDK for entertainment applications, then you don’t need to worry about any of the above, as the Hue EDK has full support for HTTPS built in.

Plain HTTP support

Previously, the Hue API V1 has been accessible over both HTTPS and plain HTTP, but support for HTTP is removed via a firmware update. The Hue API V2 has always supported HTTPS exclusively. So HTTPS must be used at all times. Furthermore, no fallback to HTTP should be implemented, since that would defeat the purpose of HTTPS (for example, a malicious actor spoofing a Hue Bridge would pose as not supporting HTTPS, triggering the fallback to HTTP to obtain sensitive data like the hue-application-key).
